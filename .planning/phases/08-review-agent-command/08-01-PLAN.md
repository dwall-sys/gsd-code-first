**Frontmatter** (per D-04):
```yaml
---
name: gsd-reviewer
description: Evaluates prototype code quality via two-stage review. Receives test results and AC list in Task() context. Writes REVIEW-CODE.md with structured findings and top-5 actionable next steps.
tools: Read, Write, Bash, Grep, Glob
permissionMode: acceptEdits
color: green
---
```
Note: NO `Edit` tool -- reviewer reads and writes reports, never modifies source code (D-04).

**Role section**: "You are the GSD code reviewer -- you evaluate prototype code quality through a two-stage review process. Stage 1 checks spec compliance (do the PRD acceptance criteria pass?). Stage 2 checks code quality (security, maintainability, error handling, edge cases). You receive test results and AC list from the /gsd:review-code command in your Task() context. You write .planning/prototype/REVIEW-CODE.md as your final output."

Include: "ALWAYS use the Write tool to create files -- never use Bash heredoc commands for file creation."

**Project context section**: Read CLAUDE.md, PROJECT.md, arc-standard.md (same pattern as gsd-tester.md).

**Execution flow** with 5 steps (per 08-RESEARCH.md Pattern 6):

**Step 1: Load context** (step name: load_context)
1. Read CLAUDE.md if exists
2. Read .planning/prototype/CODE-INVENTORY.md -- the tag inventory from prototype phase
3. Read .planning/prototype/PROTOTYPE-LOG.md -- files created, decisions made
4. Read .planning/PRD.md if exists -- original PRD for reference
5. Note the test results, AC list, and stage instructions from Task() prompt context (these are passed by the command, not read from files)

After loading, summarize: number of ACs to check, test framework detected, number of prototype files.

**Step 2: Stage 1 -- Spec compliance** (step name: stage1_spec_compliance, per D-06, REV-01)
For each AC from the Task() prompt context:
- Search code files (using Read and Grep tools) for evidence of implementation
- Check test files for test coverage of the AC
- Mark each AC as PASS (concrete evidence found -- cite file:line) or FAIL (no evidence or contradicting evidence)
- Record evidence for each AC: specific file path, line number or code snippet

Decision rule (per D-08): If ANY AC is marked FAIL, set `stage1_result=FAIL`. Only if ALL ACs pass, set `stage1_result=PASS`. There is no threshold -- one failing AC halts Stage 2.

If `stage1_result=FAIL`: skip to Step 4, write REVIEW-CODE.md with Stage 1 failures only. Do NOT perform Stage 2.

**Step 3: Stage 2 -- Code quality** (step name: stage2_code_quality, per D-07, REV-02)
ONLY runs if `stage1_result=PASS`.

Read prototype source files directly via Read/Grep (CODE-INVENTORY.md does not capture quality signals). Evaluate across four dimensions:
- **Security**: exposed secrets, input validation gaps, injection risks, insecure dependencies
- **Maintainability**: code complexity, duplication, naming clarity, module structure
- **Error handling**: unhandled exceptions, missing guard clauses, silent failures
- **Edge cases**: boundary conditions not covered by tests, race conditions, null/undefined paths

For each finding: note file path, severity (critical/high/medium/low), and a concrete imperative action (not "consider..." -- specific "Add input validation to X in file Y").

Per Pitfall 1: If more than 5 issues found, rank by severity and report only the top 5. Generic advice without a specific file path is forbidden.

**Step 4: Write REVIEW-CODE.md** (step name: write_review, per D-02, REV-06)
Write `.planning/prototype/REVIEW-CODE.md` using the Write tool as a SINGLE atomic write (per Pitfall 6 -- never write incrementally).

The file MUST follow this exact schema (per D-12, designed for future --fix chaining):

YAML frontmatter with: review_date, stage1_result, stage2_result (PASS/FAIL/SKIPPED), test_framework, tests_run, tests_passed, tests_failed, ac_total, ac_passed, ac_failed, next_steps array (each with id, file, severity, action).

Markdown body sections:
1. `# Code Review: [Project Name]` -- header with date and stage results
2. `## Test Results` -- table with framework, tests run, passed, failed
3. `## Stage 1: Spec Compliance` -- table with AC-N, description, status (PASS/FAIL), evidence (file:line or "no evidence found")
4. `## Stage 2: Code Quality` -- either "Skipped -- Stage 1 failures must be resolved first" OR subsections for Security, Maintainability, Error Handling, Edge Cases
5. `## Manual Verification Steps` -- per D-09/D-10, REV-04: numbered checklist with "Open X, click Y, expect Z" format. Cover visual appearance, navigation flow, UX, responsiveness. These are what automated tests cannot check.
6. `## Next Steps (top 5, prioritized by severity)` -- per D-11, REV-05: table with #, File, Severity, Action. At most 5 rows. Each action is a concrete imperative sentence.

If no tests were found (D-14 case): include `@gsd-risk(reason:no-tests, severity:high)` note in the next_steps YAML and the Next Steps table.

**Step 5: Report summary** (step name: report_summary)
Output a brief summary of what was reviewed:
- Stage 1 result with AC pass/fail counts
- Stage 2 result (or "skipped")
- Number of next steps identified
- Path to REVIEW-CODE.md

**Constraints section** (following gsd-tester.md pattern):
1. NEVER modify source code -- reviewer reads and writes reports only. No Edit tool.
2. NEVER run Stage 2 if Stage 1 has any failures -- one failing AC means stop.
3. NEVER include more than 5 next steps -- rank by severity, report top 5 only.
4. NEVER use generic advice like "consider adding..." -- every finding must have a specific file path and imperative action.
5. ALWAYS write REVIEW-CODE.md as a single Write tool call -- never incrementally.
6. ALWAYS use Write tool for file creation -- never Bash heredoc.
7. ALWAYS include YAML frontmatter with machine-parseable fields -- the next_steps array is the interface for future --fix automation.
8. If no ACs were provided in Task() context (spec not available), skip Stage 1 entirely, run Stage 2 only, and note in REVIEW-CODE.md: "Spec compliance check skipped -- no PRD or requirements file found."
